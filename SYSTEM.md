# 「プロジェクト・フール」システム仕様書（SYSTEM.md）

本仕様書は、メタ構造・日常侵食型ノベルアドベンチャーゲーム「プロジェクト・フール」のシステム、技術構成、およびスマートフォン向け最適化の設計仕様について記述します。

---

## 1. ターゲットプラットフォームと動作環境

本作は、手軽にアクセスして「日常が侵食される」体験を演出するため、**スマートフォンのWebブラウザを最優先のメインターゲット**として開発します。

*   **動作ターゲット**:
    *   スマートフォン（iOS/Android）の標準ブラウザ（Safari, Google Chrome）
    *   PCブラウザ（Safari, Chrome, Edge, Firefox）でのレスポンシブ表示も完全対応
*   **配信形態**: 静的HTMLホスティング（GitHub Pages, Netlify, Vercel等）によるWeb公開

---

## 2. 技術スタック

メンテナンス性、軽量化、およびメタギミックの自由度を両立するため、以下のスタックを使用します。

*   **基盤フレームワーク**: **Astro (v4.x)**
    *   高速な静的サイトジェネレート（SSG）とコンポーネント管理
    *   Viteによる快適なローカル開発環境（HMR対応）
*   **ユーザーインターフェース**: **HTML5 & Vanilla CSS**
    *   高速描画と滑らかなアニメーション制御のためのCSS VariablesおよびKeyframeアニメーションの多用
    *   TailwindCSS等のユーティリティは使用せず、柔軟なグリッチ表現や3DフリップのためにピュアなCSSでデザイン
*   **ゲームロジック**: **Vanilla JavaScript (ES6+)**
    *   ブラウザ環境そのものをメタ的に利用するため、コードの可読性をある程度保ち、過度な難読化を避けた設計
*   **画像アセット**:
    *   ライダー・ウェイト版（1909年）のパブリックドメインのタロットカード画像を事前にローカルダウンロードし、`public/images/cards/` 配下に静的アセットとして配置してゲーム内でロード（オフライン対応および読み込みの最適化）

---

## 3. スマートフォン向け最適化（タッチファースト設計）

スマホの狭い画面とタッチUIにおいて、快適な操作とメタ体験を両立するための独自システム仕様です。

### 3.1 独立した「マルチシーン（ビュー）」による画面構成の明確化
*   ゲームのコンテキストや演出に合わせて、画面全体を4つの独立したビュー（Talk:占い店対面, Chat:アプリトーク画面, Celtic:ケルト十字儀式, Puzzle:意思決定パズル）にクラス制御で瞬時に切り替えるルーティングシステムを実装しています。
*   これにより、占い師の肖像画が不要なシーンで常駐することを防ぎ、スマートフォンの画面効率を最大化します。

### 3.2 縦画面（ポートレート）に特化したレスポンシブUI
*   CSSの Flexbox と CSS Grid を用い、スマートフォンの画面アスペクト比（縦長）に自動最適化します。
*   チャットビューでは、メッセージが自動で下にスクロールされ、下部のインタラクションゾーンがキーボードや入力の阻害にならないよう最適配置されています。

### 3.3 ポインターイベントによるドラッグ＆ドロップ（カード破棄・石碑ギミック）
*   マウスとタッチパネル双方の入力を抽象化するため、**Pointer Events API**（`pointerdown`, `pointermove`, `pointerup`）を使用します。
*   これにより、スマホの画面スワイプやPC of ドラッグ＆ドロップでも、ラグのない直感的な「カードをゴミ箱に捨てる」メタギミックを共通のコードで動作させます。

### 3.4 スマホで完結する「象徴石碑ドラッグによる自律決定ギミック」
*   2周目の女教皇の試練において、単純なテキスト選択肢を廃止し、カードの周囲に散らばる「象徴の石碑（内省の鏡 / 他者拒絶）」をカードに直接ドラッグ＆ドロップして重ねることで、意思決定を行うゲームデザインを採用しています。
*   文字を入力させることなく、スマホのタッチ＆ドラッグ操作のみで「自分の意志で運命を動かした」という直感的なフィードバックと謎解き体験を両立させます。

## 4. ゲーム状態管理と永続化（セーブ）

プレイヤーの「ループ体験」を裏付けるための状態管理仕様です。

*   **メモリ上の状態（State）**:
    *   `currentLoop` (周回数: 1〜3+)
    *   `currentStep` (各周回内の進行シナリオインデックス)
    *   `isFateControlled` (運命支配フラグ: true=支配、false=自由)
*   **永続化仕様（セーブシステム）**:
    *   ブラウザの **LocalStorage** を使用して、プレイヤーがブラウザを閉じたりゲームを強制終了（リロード含む）したりしても、「これまでに何回ループしたか」「どのエンディングを見たか」の記憶を自動保持します。
    *   これにより、「ゲームをリセットしてやり直したはずなのに、占い師がプレイヤーの過去の行動を覚えている」といった認知ハッキング演出を実現します。

## 5. 最近のシステム拡張・変更点

### 5.1 話者タグパーサと動的スタイル制御（Talkビュー）
*   **状態変数 `activeDialogueSpeaker` の導入**:
    *   シナリオデータ（Markdownファイル）のページテキストの先頭に配置された `[speaker:話者名]` を正規表現（`/^\[speaker:([^\]]+)\]\r?\n?([\s\S]*)$/`）で検知し、話者名を更新します。
    *   話者タグがないページでは `activeDialogueSpeaker` が維持され、冗長なタグ記述なしに会話が進行します。
*   **`.speaker-player` クラスによる動的スタイル変更**:
    *   `updateSpeakerVisibility` 関数において、話者名が「プレイヤー」の際に、ダイアログボックス（`.dialogue-container-talk`）に対して `.speaker-player` クラスを着脱します。
    *   CSS側で `.dialogue-container-talk.speaker-player` スタイルを定義し、背景色を青みがかった暗色（`rgba(20, 25, 45, 0.95)`）に、枠線を青色（`rgba(59, 130, 246, 0.4)`）に変更します。

### 5.2 空 `desc` 選択肢による自動進行（Chat / Talkビュー共通）
*   **チャットおよび対話（Talk）画面の自動ステップ遷移**:
    *   `handleQuizChoiceSelected` 内にて、選択されたカードの `desc` が空（偽値）であるかを判定します。
    *   **Chatビュー（`view: chat`）の場合**:
        *   `!card.desc` の場合、チャット応答の吹き出しおよび「トークを進める」ボタンを生成せず、プレイヤーの行動決定メッセージ描画のタイピング完了から1秒のディレイ（`setTimeout`）後に直接 `advanceGame()` を呼び出して次のステップへ遷移させます。
    *   **Talkビュー（`view: talk`）の場合**:
        *   `!card.desc` の場合、結果テキストを表示するための `showPaginatedText` を呼び出さず、その場で直接 `advanceGame()` を呼び出して次のステップ（例：ソフィアの部屋への入店など）へ自動遷移させます。

### 5.3 デイリー・トランジション（暗転＆日付表示）演出
*   **ステップロード時の日付検知とアイキャッチ生成**:
    *   `executeLoadStep` 内にてアルカナ名（`stepData.arcana`）から `〇日目` を正規表現で抽出。
    *   該当する場合は、画面全体を覆う `day-transition-overlay`（`#05040a`）を動的に生成し、中央に日付とアルカナ名（例: "2日目", "I : THE MAGICIAN"）をフェードイン＆ズームアニメーションで表示するアイキャッチ演出を導入しました。
    *   トランジションが完了するタイミングと同期してテキストのタイピングが開始されるように制御し、別の日になったというイメージを直感的に分かりやすくしています。

### 5.4 動的背景画像変更タグ（`[bg:画像名]`）
*   **文脈に応じた背景変更のサポート**:
    *   シナリオ内に `[bg:画像名]` を記述することで、対応する背景画像（`home_morning.png`、`modern_office.png`、`city_street.png`等）へ切り替えるパーサーを `showNextDialoguePage` および `showPaginatedText` に実装。
    *   検出されたタグは背景切り替え処理を実行した上で、画面表示用テキストから自動的に削除されます。

### 5.5 チャット内選択肢結果（`desc`）のアプリ外表示化とインデックス不整合修正
*   **選択肢結果のノベル画面（Talkビュー）表示**:
    *   チャット内の選択肢の決定後、吹き出しではなく自動的に `talk` ビューに切り替えて、結果テキスト `desc` をナレーションとしてタイピング表示するように `handleQuizChoiceSelected` を変更。
*   **改ページ進行インデックス不整合の修正**:
    *   チャット移行時にノベルのページ送りインデックス（`currentDialoguePageIndex`）を末尾（`currentDialoguePages.length - 1`）に強制進めることで、結果表示後にクリックした際に以前のチャット前テキストが再表示される不整合バグを解消しました。

### 5.6 ソウルカード算出時の派手なサイバー数秘術アニメーション
*   **ビジュアル演出フロー**:
    1.  **ブラー回転とロック**: ユーザー生年月日の数字スロット（`calc-digit-slot`）が最初はランダム数字で高速回転（`blur-active` でブラーと残像）。1桁ずつ `settled-flash` で拡大フラッシュしながら順次ロック。
    2.  **数字のフライト（射出・パーティクル）**: ロックされた順にスロットの座標から計算式エリアの該当位置（`getBoundingClientRect` の相対差分）へ、光の尾を伴うクローン数字が飛び出す。
    3.  **着弾と累積加算**: 目標座標に達した瞬間、波紋スパーク（`.calc-spark-ring`）を拡散させ、計算式を徐々に表示し、その時点の合計を `ACCUMULATED VALUE` HUDでカウントアップ表示。
    4.  **警告と桁の衝突スライド合体**: 一次合計が 22 以上の時、`calc-warning-banner` のエラー警告を表示後、桁の数字（例: `3` と `3`）を左右に分離し、中央へ衝突させフラッシュと同時に `3 + 3 = 6` へ合体。
    5.  **魔法陣高速回転とゴールドフラッシュ**: 最終値決定後に魔法陣（`.calc-magic-circle`）を `fast-rotate` で超高速回転させ、ソウルナンバーを巨大な金色の脈動文字として表示。その後全画面の `gold-flash-overlay` を閃光させて結果カード表示にトランジション。

### 5.7 ケルト十字スプレッド（Step 11）の完全自動一括展開化（スマホ対応）
*   **操作性・テンポ改善仕様**:
    *   山札のクリックや、1枚ごとの手動ドロー/確認操作を完全に廃止しました。
    *   導入セリフ表示後の画面タップにより、自動展開アニメーション `startAutomaticCelticSpread()` が発火。450msの間隔で10枚のカードが自動かつ連続して山札から引き出され、盤面スロット（1〜10）へと移動・配置されます。
    *   カード展開中はモーダルのポップアップや個別の解説表示を挟まず、カードがスロットに吸い込まれていくアニメーション演出に集中させます。
    *   10枚すべての配置完了後に、ソフィアによる完成盤面の全体総括セリフを流し、その読了後のタップで契約フォーム表示 ➔ 契約完了 ➔ 次ステップ進行へ連動します。

### 5.8 テキストウィンドウのレイアウト向上とスマホ向けレスポンシブ最適化
*   **美観と表示領域の最適化**:
    *   `.text-content` でのブラウザ標準のスクロールバーをCSSで完全に非表示化しました。
    *   `dialogue-container-celtic` のスタイルを Talk ビューと同一（浮遊パネル、十分な縦幅、枠線）に統一し、ウィンドウの窮屈さを解消しました。
    *   `@media (max-width: 768px)` のメディアクエリ内で、ケルト十字盤面コンテナ（`.celtic-cross-container`）の最大高さを `52vh` に制限。これにより、ダイアログ領域が下に押しつぶされず、Talk ビューと同様の高さ（最小高さ120px）がスマホ画面下部に美しく確保されます。
    *   画面クリック時の `SKIP_SELECTORS` 判定をケルト十字ビュー内でバイパスし、盤面のカードやデッキの上をタップした場合でも、進行用イベントが透過して受け取られるように修正しました。

## 6. 解決済みのシステム不具合

### 6.1 ケルト十字スプレッド（Step 11）におけるクリック進行不能バグ 【修正完了】
*   **原因**: 
    *   スプレッドのロード時に呼び出される `highlightSlotGuide` 関数が `app.js` 内に定義されておらず、`highlightSlotGuide(1)` の実行時点で `ReferenceError` になりスクリプトが強制停止。以降の山札（`DECK`）へのクリックイベントの登録処理が実行されていませんでした。
*   **対策**:
    *   `app.js` に `highlightSlotGuide` 関数を追加定義し、対象スロットに `.active-slot` クラスを着脱するように修正。
    *   同時に、親コンテナである `.celtic-cross-container` に `position: relative` を指定していなかったために絶対配置されたカードが画面全体の基準にずれる問題をCSS側で修正しました。
